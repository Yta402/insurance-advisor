# Insurance 项目开发指南（防跑偏锚点）

> 本文件是 insurance 项目的开发指令，每次工作必须对照，防止数据收集和代码生成时目标跑偏。
> 继承根目录 AGENTS.md 的"互联网大厂 AI 产品经理 + 批判性视角的面试项目指挥官"人设与原则。

## 一句话定位

面向 20-60 岁、因未来健康问题担忧而需要保险保障的人群，基于其收入/健康/家庭条件，提供健康保障保险的**个性化方案对比 + 公司服务质量数据分析**，辅助其自行决策。

## 核心需求（不可动摇）

- **真实需求锚点**：用户对未来健康风险的担忧 + 保险信息不对称（产品条款复杂、公司服务质量不透明）
- **产品本质**：决策辅助工具——呈现对比分析，**不替用户决定**
- **双能力**：AI 个性化匹配 + 数据分析（公司×险种服务质量）

## 目标人群

- **20-60 岁**当代人，有未来健康风险担忧
- **决策三维度**：收入（预算）+ 健康情况（适配）+ 家庭条件（责任）
- 典型画像：
  - **20-30 岁**学生/职场新人：预算紧 → 医疗 + 意外（便宜基础保障）
  - **30-45 岁**家庭支柱/高压职场：→ 重疾（含返还型，小姨画像）
  - **45-60 岁**：健康担忧加剧，重疾可能保费倒挂 → **诚实劝退**，导向医疗 + 意外

## 双能力定位（简历价值）

- **AI 个性化方案匹配**（呼应简历调研 Agent 人设）
- **公司×险种服务质量数据分析**（数据分析王牌，核心亮点）

## MVP 范围（严防蔓延）

| 纳入 | 形态 |
|---|---|
| ✅ 百万医疗险 | 标准报销型 |
| ✅ 意外险 | 给付 + 报销 |
| ✅ 重疾险 | 消费型 / 储蓄型 / 返还型（含小姨案例） |

| 排除 | 理由 |
|---|---|
| ❌ 理财型（年金/增额终身寿） | 逻辑根本不同（金融精算），会让产品失焦 |
| ❌ 财险（车险/家财） | 不同逻辑/监管，车险已排除 |
| ❌ 定期寿险 | 条款简单、纠纷率低，展示不出核心差异化 |
| ❌ 小额医疗/高端医疗 | 同质化/窄众，分析价值低或数据量小 |

## 合规红线（绝不触碰）

1. ❌ 不做保险销售 / 代销 / 投保
2. ❌ 不做"推荐购买某款"，只做"对比呈现"
3. ❌ 不代核保 / 不代答健康告知 / 不根据健康推荐具体产品
4. ✅ 健康情况只做"适配提示"（展示健康告知宽松度），不做核保判断
5. ❌ 不做精算定价 / 理赔结论
6. ✅ 全程"对比/呈现/辅助"措辞，决策权在用户

## 数据收集红线

- **两个数据源**：
  - 产品费率表（条款 + 保费矩阵：产品 × 年龄 × 性别 × 保额）
  - 金融监管总局（原银保监）保险消费投诉情况通报 + 行政处罚公告
- **可得性是项目生死线**，拿不到则废案，不糊弄
- 投诉通报必须按**险种粒度**公开（重疾 vs 医疗 vs 意外），否则"公司×险种"差异化打折
- 公开数据优先，合规采集

## 技术栈

- AI：DeepSeek / Kimi / GLM（已有 key）
- 数据库：SQL（PostgreSQL 或 SQLite）
- 数据分析：SQL + Pandas
- 后端：FastAPI
- 前端：Streamlit
- 可视化：Plotly

## 防跑偏检查清单（每次开发对照）

- [ ] 是否在做"对比分析"而非"推荐销售"？
- [ ] 数据分析是否是核心（公司服务质量）而非配角？
- [ ] 是否聚焦 3 险种（医疗 + 意外 + 重疾），无范围蔓延到理财/财险？
- [ ] 是否守住合规边界（不代核保、不推荐具体产品）？
- [ ] 是否面向 20-60 岁健康保障场景，没有跑偏？
- [ ] 是否诚实处理高龄段（45-60 岁重疾保费倒挂要劝退）？
