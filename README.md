# Insurance Advisor · 保险决策数据辅助工具

> 面向 20-60 岁、对未来健康问题担忧的人群，基于**公开监管数据**，提供保险公司服务质量分析 + 个性化险种方案参考。
> **定位：决策辅助工具——客观呈现，不构成投保建议。**

## 为什么做这个

买保险存在严重**信息不对称**：产品条款复杂难懂、保险公司服务质量不透明。现有渠道（小红书攻略、卖保险平台）要么是静态通用攻略，要么利益相关。本工具用**监管投诉数据**做"公司×险种服务质量"的中立量化分析——这是现有方案做不到的。

## 产品流程

```
用户画像(年龄/收入/家庭/健康/预算)
   ↓ LLM 推理
适合的险种组合 + 保额区间（AI 匹配）
   ↓ SQL 筛选
符合条件的产品（待产品库扩充）
   ↓ 关联
各产品对应公司的投诉/服务质量数据（数据分析核心）
   ↓ 综合呈现
辅助用户自行决策（不替用户决定）
```

## 技术选型（批判性落实：任务本质决定技术）

| 环节 | 任务本质 | 选型 | 不用 X 的理由 |
|---|---|---|---|
| 通报→公司+数值抽取 | 半结构化公文，数值需 100% 准 | **正则主抽 + LLM 仅兜底公司名** | 纯 LLM 数字幻觉致命 |
| 公司分类(人身/财险) | 有限类+强关键词信号 | **关键词规则 + 字典** | LLM 杀鸡用牛刀 |
| 统计分析(排名/趋势) | 描述性统计 | **SQL** | LLM 算均值又慢又可能错 |
| 产品条款要素抽取 | 复杂自然语言理解 | **LLM**（真 AI） | 正则必崩，待产品数据 |
| 用户画像→险种推荐 | 多约束推理+个性化解释 | **LLM**（钉合规边界） | 规则组合爆炸 |
| 险种→具体产品匹配 | 结构化条件筛选 | **SQL** | 不需推理 |
| BM25 / Embedding | — | **不用** | 无海量检索/语义匹配场景 |
| 预测模型 | — | **不做** | 伪需求（消费者不关心未来预测、数据太少不可靠） |

**原则**：不为显得高级而堆技术。数值抽取禁纯 LLM（幻觉红线）；简单任务用规则/SQL；LLM 只用在语义理解/推理不可替代处。

## 数据源与局限（诚实声明）

| 数据 | 来源 | 状态 | 局限 |
|---|---|---|---|
| 保险公司投诉 | 金融监管总局《保险消费投诉情况通报》 | ✅ 已采 3 季度(2022Q3/Q4/2023Q1)，160 条 | 通报仅列"投诉较高"前若干家，**非全市场**；"未上榜"≠"无投诉"；季度扩充受限于官网检索 |
| 保险产品条款/费率 | 各公司官网公开信息披露 | ⏸ 待采集 | 行业协会 icidp 声明禁止提取，改走各公司官网 |

## 合规边界（绝不触碰）

- ❌ 不销售 / 代销 / 投保
- ❌ 不推荐购买某款，只做对比呈现
- ❌ 不代核保 / 不代答健康告知 / 不根据健康推荐具体产品
- ✅ 健康情况只做"适配提示"，不做核保判断
- ✅ 全程"对比/呈现/辅助"措辞，决策权在用户

## 运行

```bash
# 环境：Python 3.11 + Playwright + plotly/pandas/openai
conda create -p D:\anaconda3\envs\insurance python=3.11 -y
pip install playwright plotly pandas openai
playwright install chromium

# 设置 DeepSeek API Key
$env:DEEPSEEK_API_KEY="your_key"

# 流程
python scripts/fetch_quarterly.py    # 采集
python scripts/extract_to_db.py      # 抽取入库
python scripts/analyze.py            # 分析
python scripts/visualize.py          # 看板
python scripts/ai_match.py           # AI 匹配
```

## 技术栈

Python · Playwright（JS 渲染采集）· SQLite（结构化存储）· SQL（描述性统计）· Plotly（可视化）· DeepSeek（LLM 推理）
